# Workflow Benchmark v18 汇报版摘要

## 1. 我们这版数据集在做什么

这版 benchmark 不再只测“单个网页任务会不会做”，而是测：

- 给定一个**高层目标**
- agent 能不能自己决定要做哪些子任务
- 能不能把这些子任务按合理顺序拼起来
- 能不能在多条可行路径里选一条完成目标

一句话说，就是从 **单题执行** 升级到了 **任务流组合**。


## 2. 几个核心名词

- **module**
  - 一个可复用的子任务模块
  - 例如“开户”“订机票”“申请退款”

- **blueprint**
  - 一类 workflow 样本的生成模板
  - 它规定：高层目标是什么、有哪些可行路径、哪些参数可以变化

- **instance**
  - 由 blueprint 具体生成出来的一道题
  - 例如把“订机票”具体实例化成“从北京飞上海，日期是 2025-12-31”

- **predicate**
  - 状态名
  - 用来表示“现在世界里已经满足了什么条件”
  - 比如 `bank_account_active`、`travel_booking_confirmed`

- **split**
  - 训练集 / 验证集 / 测试集划分
  - 我们不是按单条 instance 划分，而是按 blueprint 划分，避免只换参数造成泄漏


## 3. 这版数据集的规模

当前正式版本是 **v18**。

- 主题数：`14`
- blueprints：`504`
- 抽象模块：`108`
- module bindings：`112`
- 共享状态词表：`149`

正式实验批次：

- `train = 4760`
- `dev = 140`
- `test = 140`
- `total = 5040`


## 4. 这版为什么比旧版更强

旧版更像是在测：

- 单个网页题能不能做对
- 或者固定 task chain 能不能顺着做下去

新版更强调：

- 同一个目标，是否有多种完成方式
- agent 是否能自己选模块
- agent 是否能自己决定顺序
- agent 是否能跨不同任务类组合模块

所以它更接近真实 agent 的使用场景。


## 5. 代表性例子

### 例子 1：新生安顿

目标：

- 准备好住房财务信息
- 开通手机服务

两条路径都能完成：

1. 路径 A
- 找房
- 做租约登记
- 开通手机套餐

2. 路径 B
- 找房
- 做租约管理复核
- 开通手机套餐

这里测的是：

- 同一个高层目标
- 可以通过两种不同的住房后续模块完成


### 例子 2：售后补救

目标：

- 已与支持系统建立联系
- 已提交保修/补救请求

两条路径：

1. 路径 A
- 联系客服
- 发起保修索赔

2. 路径 B
- 走物流修复
- 发起保修索赔

这里测的是：

- 在“订单已经送达且出现问题”的前提下
- 可以通过“客服升级”或“物流修复”两种不同售后策略完成
- 而不是不合理地从重新下单开始


### 例子 3：安全加固

目标：

- 访问面已审查
- 安全加固已完成

两条路径：

1. 路径 A
- 直接配置密码管理器

2. 路径 B
- 开启 2FA
- 检查隐私设置
- 做密钥轮换

这里测的是：

- 同一个目标
- 可以靠一个高价值模块直接完成
- 也可以靠多个细粒度模块组合完成


### 例子 4：旅行准备

目标：

- 行程已确认
- 出行许可已确认

两条路径：

1. 路径 A
- 订机票
- 检查签证/许可要求

2. 路径 B
- 直接走长途旅行模块

这里测的是：

- agent 是不是会选择一条更短的复合路径
- 或者自己显式组合多个子模块


### 例子 5：跨任务类组合

目标：

- 支付准备完成
- 配送状态已确认

两条路径：

1. 路径 A
- 开户
- 购物
- 配置复杂自动扣款
- 确认订单到达

2. 路径 B
- 开户
- 转账
- 联系客服确认配送状态

这个例子最能说明新版 benchmark 的特点：

- 它不是只在旧的某一个任务类里做 chain
- 而是允许跨不同任务类组合模块


## 5.5 理想运行方式

这套 benchmark 理想上是交互式运行的，不是一次性把整条路径都告诉模型。

流程是：

1. 先把一个高层目标样本给模型
2. 模型先选择下一步要做的抽象模块（module）
3. 系统再把这个模块映射到一个具体子任务（binding）
4. 子任务实例化后，再交给模型去完成页面交互
5. 执行完以后更新世界状态，再进入下一轮模块选择

也就是说，这里同时在测两层能力：

- 高层 workflow 规划能力
- 低层网页任务执行能力

当前我们已经有了最小可用的 runner 原型：

- `/Users/masteryth/Documents/webagent/rl_memory/scripts/run_workflow_episode.py`

它已经能先完成：

- `goal/oracle -> reference path -> module -> binding -> atomic subtask -> execution trace -> episode evaluation`

因此现在不是只有数据集设计，已经有了一条可运行的 workflow episode 原型链路。


## 6. 质量控制

我们不是只追求数量，还加了硬质量门槛。

目前 `v18` 满足：

- `subset-like multi-path = 0`
  - 含义：不会出现“一条路径只是另一条路径多塞一个无关模块”这种假多路径

- `initial-target-overlap = 0`
  - 含义：不会出现“初始状态本来就已经包含目标状态”的脏样本

- `hard_fail_reasons = []`
  - 含义：严格审计没有发现结构性错误


## 7. 当前可以怎么讲

一个比较稳的讲法是：

> 我们把 benchmark 从 atomic web tasks 提升到了 compositional workflow goals。  
> 在 v18 里，我们固定了 14 个主题，构建了 504 个 workflow blueprints，并生成了 5040 条实例。  
> 这些样本不是固定 task chain，而是同一高层目标下存在多条语义上不同的完成路径。  
> 同时我们通过 blueprint-level split 和严格质量门槛，尽量避免只换参数带来的数据泄漏和假多路径问题。


## 8. 当前结论

这版数据集已经到达可以直接开实验的阶段：

- 规模够大
- 主题够全
- 多路径比例高
- 质量门槛已经硬化
